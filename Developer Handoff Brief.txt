Bloom Culture DIY Wedding Flower Planner — Developer Handoff

Goal: Take this finished front-end app and launch it on bloomcultureflowers.com (Shopify) as a private, customer-only tool where:

1. Anyone who pays gets access — whether they bought wedding flowers *or* the standalone planner product. (All paid orders grant access.)
2. Customers log in and their planner data is saved to their account, so "their wedding" follows them across devices (phone, laptop, etc.).
3. The planner can also be sold on its own as a product.

The app itself is complete and tested. Your job is auth, payment-gating, and swapping the storage layer from browser-local to per-customer cloud storage. Estimated 15–30 hours.

Owner's finalized decisions (read first)
• Access window: access is granted after any paid order and expires 2 years from the customer's MOST RECENT qualifying order (a new qualifying purchase resets the 2-year window). Enforce by validating the qualifying purchase date at access time (when the customer opens the planner) — do NOT rely on scheduled jobs / periodic tag cleanup.
• Immediate access + confirmation email: the experience must be: order → instantly tagged with access → order confirmation email includes an "Access your DIY Flower Planner →" button/link → customer clicks and is in. Customize the Shopify order confirmation email for any qualifying order, and ensure customer accounts are enabled at checkout so the planner saves and follows the customer across devices.
• Image-slot storage: use Supabase Storage (uploaded assets + stored URLs). Do not use base64/jsonb for images.
• No existing gating app — the access flow is net-new.
• Email platform is Flodesk (not Klaviyo) — any transactional/notify hooks should account for that.
• Installed apps to respect: Simple Bundles & Kits, Shopify Flow (active), Puco Checkout Validation (may affect checkout), Appointo, Bird Pickup Delivery, Zapier, Agentic. Customer accounts are standard Shopify (not Plus/B2B).
• Theme: SS 8.0 Broadcast with custom Liquid/CSS — place the landing page via a dedicated page template (see §0), do not disturb existing sections.

⭐ CRITICAL ARCHITECTURE REQUIREMENT — single hosted source

Host the app in ONE place; the website loads it from there. When the app is updated at that source, the live site reflects the new version automatically — with NO re-deploy, re-paste, or developer involvement.
• Do NOT inline/paste a static copy of the app into a Shopify page or template (that creates a frozen copy that must be manually re-pasted on every change — unacceptable).
• Instead, host the app's files at a single location (e.g. a CDN / static host / subdomain like planner.bloomcultureflowers.com, or Shopify-hosted assets referenced by URL) and have the Shopify page load it from there (script/iframe/embed).
• The non-technical owner must be able to update content (the app's files at the source) and have every customer instantly see the update — without paying for a dev each time.
• This applies to the landing page too where feasible: prefer loading from the single hosted source over a pasted copy.

This is a hard requirement, not a preference. Architect everything around it.

2. The two things to build

A) Access control (Shopify)
ALL paying customers get in. The simplest reliable pattern:

1. Shopify Flow (free app) → one workflow:
• Trigger: Order paid
• Action: Add customer tag planner-access
• This fires for *every* paid order — flowers, the standalone planner, anything. Requirement #1 done, with no per-product config.
2. Standalone planner product: create a normal Shopify product ("DIY Wedding Flower Planner"). Because the Flow above tags on *any* paid order, buying just the planner also grants access automatically.
3. Gate the planner page so only customers with the planner-access tag can view it. Two options:
• No-code: the Locksmith app (~$9/mo) → lock the planner page to customers tagged planner-access.
• Code: check customer.tags in Liquid on the planner page/template and redirect un-tagged visitors to a "purchase to unlock" page.

Login = Shopify's native customer accounts (already enabled for their store). The logged-in customer object is what we key data to.

B) Per-customer cloud storage (the one real code change)
Right now all state saves to localStorage. It must save to a per-customer record instead so it syncs across devices.

The entire storage layer is isolated in components.jsx — these three functions are the *only* place data is read/written:

``js
const LS_PREFIX = "bcf.v1.";
function lsGet(key, fallback){ /* localStorage.getItem */ }
function lsSet(key, val){ /* localStorage.setItem */ }
function useLocal(key, initial){ /* React hook wrapping the two above */ }
`

Every checklist, note, worksheet, date, and palette choice flows through useLocal(key, initial). Image slots (image-slot.js) persist separately — see §4.

Your task: replace lsGet/lsSet with calls to a per-customer store, keyed by Shopify customer ID. Recommended: Supabase (free tier is plenty; Postgres + auth + JS client).

Minimal shape:
`
Table: planner_state
  customer_id   text   (Shopify customer ID)
  key           text   (the bcf.v1.* key)
  value         jsonb
  primary key (customer_id, key)
`
• On load: fetch all rows for customer_id, hydrate an in-memory map, have lsGet read from it.
• On change: lsSet upserts {customer_id, key, value} (debounce ~500ms).
• Keep localStorage as an offline fallback/cache if you like, but the server record is the source of truth.

Tip: keep the function signatures identical (lsGet(key, fallback), lsSet(key, val), useLocal(key, initial)) so nothing else in the codebase needs to change. It's a drop-in swap.

3. Wiring auth → the app

The app currently shows a mock login gate (Gate component in app.jsx) that accepts any email and sets entered=true in localStorage. Replace that with real auth:

1. The planner page only renders for logged-in, tagged customers (gate from §2A).
2. Pass the Shopify customer ID + email into the app (e.g. inject a window.BC_CUSTOMER = { id, email } via Liquid on the page, or read it from the Supabase session).
3. In app.jsx, remove/replace the mock Gate: if window.BC_CUSTOMER exists, skip straight to the portal; use its id as the storage key prefix in the new lsGet/lsSet.

Two reasonable architectures (pick one):
• Simplest: Shopify customer login + Liquid tag-gate handles access; inject customer ID into the page; Supabase stores data keyed by that ID. (Least moving parts.)
• Supabase Auth end-to-end: mirror Shopify customers into Supabase Auth and let Supabase manage sessions + row-level security. (Cleaner security; a bit more setup.)

Check image-slot.js for its save/load hooks; same idea as §2B.

5. Hosting the app on Shopify

Because there's no build step (React via CDN), you have flexibility:
• Inline option: run the bundler to produce a single self-contained .html and paste it into a Shopify page/template. Simple, but harder to update.
• Recommended: host the (small) static files on the same domain (Shopify CDN assets, or a subdomain like planner.bloomcultureflowers.com via Netlify/Vercel/Cloudflare Pages) and embed/redirect from the gated Shopify page. Easier to push updates.

Either way the access gate (§2A) controls who reaches it.

7. Notes & gotchas
• Keep the storage function signatures identical — the swap is meant to be contained to ~lsGet/lsSet/useLocal + image-slot hooks. Don't refactor the modules.
• All saved keys are prefixed bcf.v1. — preserve that namespacing per customer.
• The app is plain React + Babel-in-browser; no node_modules to install. Open index.html to run it.
• Brand colors, fonts, and content all live in index.html (CSS variables) and content.jsx — non-developers can edit copy in content.jsx` safely.
• There is no sensitive logic in the front end to protect; access control is enforced by the Shopify gate + (ideally) Supabase row-level security, not by hiding the HTML.

Stack summary for a quote: Shopify (customer accounts, Flow, + Locksmith or Liquid gate) + Supabase (Postgres + client, optionally Auth) + swap one storage module in a finished React app. No app build pipeline.
